Runtime

Runtime

  • objective-C的动态性是由runtime API来支撑的
  • runtime API提供的接口基本都是C语言的，源码又C\C++\汇编语言编写

isa

  • 在arm64架构之前，isa就是一个普通的指针，存储着Class、Meta-Class对象的内存地址
  • 从arm64架构开始，对isa进行了优化，变成了一个共用体（union）(结构体和数据共用的一个内存地址，实际共用体中的结构体，只做展示作用)结构，使用位域来存储更多信息

isa结构

  1. 共用体（union）结构(结构体和数据共用的一个内存地址，实际数据存储在bits中，共用体中的结构体，只做展示作用)

isa指针每位的含义

从内存地址，从上往下，即地址从右边，往左

  • nonpointer
    • 0,代表普通的指针，存储着Class、Meta—Class对象的内存地址
    • 1,代表优化过，使用位域存储更多的信息
  • has_assoc
    • 是否又设置过关联对象，如果没有，释放会更快
  • has_cxx_dtor
    • 是否有C++的析构函数（.cxx_destruct),如果没有，释放会更快
  • shiftcls
    • 存储着Class、Meta-Class对象的内存地址信息
    • ios中mask为0x0000000ffffffff8,因此，很多类以及元类的地址是以8和0结尾的，因为最后三位均是000.
  • magic
    • 用于在调试时分辨对象是否未完成初始化
  • weakly_referenced
    • 是否有被弱引用指向过，如果没有,释放时会更快
  • deallocating/unused(818版本之后)
    • 对象是否正在释放
  • extra_rc
    • 里面存储的值是引用计数器减1
  • has_sidetable_rc
    • 引用计数器是否过大无法存储在isa中，如果为1，那么引用计数会存储在一个叫SideTable的类的属性中

注意，这里的都是是否有设置过，就像关联对象，就算最后将所有的关联对象删除，仍是1，如果有这些，在销毁的时候，会做额外工作

当block内部访问了对象类型的auto变量时

  • 如果block是在栈上，将不会对auto变量产生强引用

  • 如果block是在堆上

    • 会调用block内部的copy函数
    • copy函数内部会调用_Block_objc_assign函数
    • _Block_objc_assign函数会根据auto变量的修饰符（_string、_weak、_unsafe_unretained）做出相应的操作，类似于retain（形成强引用、弱引用）

      block的内存管理

  • 当block在栈上时，并不会对__block变量，以及auto的对象产生强引用

  • 当block被copy到堆时

    • 会调用block内部的copy函数
    • copy函数内部会调用_Block_object_assign函数
    • _Block_object_assign函数会对_block变量形成强引用,会区分是auto对象（会区分是_weak还是_strong），还是_block对象（强引用）

  • 当block从堆移除

    • 会调用block内部的dispose函数
    • dispose函数内部会调用_Block_object_dispose函数
    • _Block_object_dispose函数会自动释放引用的_block变量
    • 当__block修饰对象时，其构建的_block变量内，也会有copy方法和dispose方法
      • 其内copy方法，同样也有_Block_object_assign,以及_Block_object_dispose,对其内修饰的对象进行内存管理（跟外围是_strong还是_weak有一定关系）
        • 但是注意，该内存管理只会在ARC下生效，如果在MRC下，在_block不会增加其内对象的引用，属于unowned关系
        • 如果不是__block,而是直接引用了auto对象，则和此前一样，block直接引用对象(mrc下，copy时，内部也会对auto增加引用)，而非通过__block间接引用

  • _block的生成的对象，通过fowarding去指向的原因

    • 在栈上的时候，指针指向自己
    • 而赋值到堆上时，栈上的指向堆上复制的，栈上赋值时，其值是保存堆上的值，从而保证了，无论是在栈上还是堆上修改，最后值都会保存的堆上。最终使用的是堆上的block

load、initialize方法的区别

2022-12-28
底层原理

load、initialize方法的区别

  1. 调用方式

    • load是根据函数地址直接调用
    • initialize是通过objc_msgSend调用

  2. 调用时刻

    • load是runtime加载类、分类的时候调用（只会调用1次）
    • initialize是类第一次发送到消息的时候调用，每个类只会initialize一次（父类的initialize方法可能会被调用多次）

  3. load、initialize的调用顺序

    1. load
      • 先调用类的load
        • 先编译的类，优先调用load
        • 调用子类的load之前，会先调用父类的load
      • 再调用分类的load
        • 先编译的分类，优先调用load
    2. initialize
      • 在，第一次使用的时候调用，先初始化父类，
      • 再初始化子类（可能最终调用的是父类的）

Load方法

  • +load方法会在runtime加载类、分类时调用
  • 每个类、分类的+load，在程序运行过程中只调用一次
    • flags标识位保证类只会加载一次

    • 同一级的关系时按照编译的顺序来
  • 尽管load中方法在源码中只会调用一次，但实际上主动[Class load]也会触发load,因此在调用method_exchangeImplementations做hook时，会加上dispatch_once，保证只会执行一次。

懒加载类以及分类的加载

2022-12-26
底层原理

官方在对类进行处理的时候, 为了提高对类处理的效率以及性能, 就对类进行了识别, 当类需要使用的时候, 系统才会对类进行实现. 如果没有使用就不会实现. 当需要实现才进行加载的类就被称为懒加载类. 反之无论是否使用到这个类, 都对这个类进行加载的类就被称为非懒加载类

  • 懒加载类和非懒加载类的区别在于是否实现了+(void)load方法
    所有的类在使用前都会调用realizeClassMaybeSwiftAndLeaveLocked对类进行加载,通过这个来判断类和分类加载的时机

底层原理

runtime初始化objc_init,会注册三个函数map_images、load_images、unmap_image,这里对map_images所做的工作进行探索,

  • 整个流程是map_images -> map_images_nolock -> _read_images。这里对map_images_nolock和_read_images进行分享

map_images_nolock

map_images_nolock源码

分析源码

  1. 判断firstTime，firstTime为YES，则执行环境初始化的准备，为NO就不执行
  2. 计算class数量，根据总数调整各种表的大小并做了GC相关逻辑处理(不支持GC则打印提示信息)
  3. 判断firstTime，firstTime为YES，执行各种表(SideTablesMap表，associations关联表,sel方法表)初始化操作，为NO则不执行
  4. 执行_read_images进行读取，然后将firstTime置为NO，就不再进入上面的逻辑了，下次进入map_images_nolock就开始直接_read_images

_read_images

_read_images源码

分析

  • 主要流程如下：
    1. 静态变量doneOnce,控制只加载一次，这里主要的工作时初始化类映射表gdb_objc_realized_classes以及重新初始化TaggePointer环境
    2. 修复预编译阶段的@selector的混乱问题,方法列表和注册时不同的，重新修复
    3. 开始遍历头文件，进行类与元类的读取操作并标记(旧类改动后会生成新的类，并重映射到新的类上),主要的方法是readClass,其内部
      • 找不到该类的父类，可能是弱绑定，直接返回nil
      • 找到类了，判断这个类是否是一个future的类(可以理解为需要实现的一个类，也可以理解为这个类是否有变化)，如果有变化则创建新类，并把旧类的数据拷贝一份然后赋值给新类newCls，然后调用addRemappedClass进行重映射，用新的类替换掉旧的类，并返回新类newCls的地址
      • 找到类了，如果类没有任何变化，则不进行任何操作，直接返回class
      • 期间会将类的信息加入NXMapTable(类映射表)和objc::allocatedClasses(所有类的表,这个表在runtime_init时初始化的)
        • addNamedClass,key是类名，value是Class,哈希表结构
        • addClassTableEntry,直接将Class和其元类加入objc::allocatedClasses类似set中，会通过这个来判断是否是isKnownClass
      • readClass会返回一个类，这里会判断，类表中的类和readClass生成的新类是否相等，如果不等，且认为新类，该类认为是Future类（在镜像加载之前就存在的类，需要重新映射，这个类有变化）
    4. 修复重映射⼀些没有被镜像⽂件加载进来的类,主要方法remapClassRef
    5. 修复⼀些旧的objc方法,在方法fixupMessageRef，类似alloc、allocWithZone:、retain、release、autorelease、objc_msgSendSuper2_fixup等一些方法，会重写赋值新的imp。在调试发现，并未触发，显然是对旧版本方法的修复
    6. 找到所有的协议protocols，和3类似，主要方法是readProtocol,会将协议加载到一个NXMapTable的映射表中，这里如果声明的协议没有使用，是不会加入的
    7. 和4类似，重映射没有从镜像中加载的协议，主要方法remapProtocolRef
    8. 分类处理，将分类合并到类中。注意,这里的合并操作，是在真正的分类合并操作完成之后（也就是在load_images内），才能进行合并，一般情况下，并不会执行
    9. 对类中有+load存在的,即非懒加载类（有+load的类）,加入到所有类的表中，并进行加载realizeClassWithoutSwift。的作用是，对类 cls 进行首次初始化，包括分配其读写数据。不执行任何 Swift 端初始化。返回该类的真实类结构。
    10. 对3种得到的FutureClasses重新映射
      • 源码
      • 非懒加载的类的realizeClassWithoutSwift是在map_images是实现的，而懒加载类则是在,第一次使用该类时，调用相关objc_msgSend方法时调用。（代码中不使用的类，则会回加载解析）
