swift派发机制

Swift派发机制分：静态派发和动态派发

  • 内联（inline）,属于静态派发的一种，最快的派发方式，在函数调用的地方，编译器直接展开函数的代码。
  • 静态派发：（又叫：直接调用）
    • 静态派发机制，同时支持值类型和引用类型；静态派发是最快的, 不止是因为需要调用的指令集会更少, 并且编译器还能够有很大的优化空间,静态派发也有人称为直接调用.
    • 结构体中以及extension的方法，默认都是直接派发
    • 然后，直接调用也是最大的局限, 而且因为缺乏动态性所以没办法支持继承
  • 消息派发：
    • 消息派发是动态派发机制的一种，仅支持引用类型(reference types) 比如：Class，对于动态性或者动态派发，我们需要使用到继承特性，而这是值类型不支持的。
    • 消息派发是调用函数最动态的方式. 也是Cocoa的基石, 这样的机制催生了 KVO,消息转发 等功能.
    • 这种运作方式的关键在于开发者可以在运行时改变函数的行为. 不止可以通过swizzling来改变, 甚至可以用isa-swizzling修改对象的继承关系, 可以在面向对象的基础上实现自定义派发.
  • 函数表派发：
    • 函数表派发是编译型语言实现动态行为最常见的实现方式. 函数表使用了一个数组来存储类声明的每一个函数的指针. 大部分语言把这个称为 virtual table(虚函数表), Swift 里称为 witness table. 每一个类都会维护一个函数表, 里面记录着类所有的函数, 如果父类函数被 override 的话, 表里面只会保存被 override 之后的函数. 一个子类新添加的函数, 都会被插入到这个数组的最后. 运行时会根据这一个表去决定实际要被调用的函数.
    • 查表是一种简单, 易实现, 而且性能可预知的方式. 然而, 这种派发方式比起静态派发还是慢一点. 从字节码角度来看, 多了两次读(读函数表地址，以及函数地址)和一次跳转, 由此带来了性能的损耗. 另一个慢的原因在于编译器可能会由于函数内执行的任务导致无法优化. (如果函数带有副作用的话)
    • 这种基于数组的实现, 缺陷在于函数表无法拓展。子类会在虚数函数表的最后插入新的函数, 没有位置可以让 extension 安全地插入函数，因此extension中的方法是不会加到函数表内的，属于静态派发

这四种派发技术，由编译器来决定使用哪种。

静态派发VS动态派发（Swift VS OC）

  • OC默认支持动态派发，这种派发形式以多态的形式为开发人员提供了灵活性。

    • 比如：子类可以重写父类的方法

  • 动态派发以一定的运行时开销为代价，提供了语言的灵活性。

  • 在动态派发机制下，对于每个方法的调用，编译器必须在方法列表中查找执行方法的实现。

  • 编译器需要判断调用方，是选择父类的实现还是子类的实现，而且由于所有对象的内存都是在运行时分配的，因此编译器只能在运行时执行检查。

  • 而静态调用则没有这个问题。在编译期的时候，编译器就知道要为某个方法调用某种实现。因此编译器可以执行某些优化，甚至在可能的情况下，可以将某些代码转换成inline函数，从而使整体执行速度更快

swift中实现动态派发和静态派发

动态派发

  • 函数表派发
    • 类中定义的方法默认是函数表派发
    • 协议中定义的方法，在调用协议时，会去协议的函数表中查询（这种查询过程，也是函数派发)，注意编器可以优化，可能会去除去协议函数表查询的过程。
  • 消息派发
    • 方法前加上@objc，OC中的代码调用swift方法，属于消息派发，但是如果是
      • class内部定义的方法，swift调用还是函数调用
      • extension内部定义的方法，则是消息派发
      • 此外，注意，一般而言extension中定义的方法，子类是无法重写的，但是由于加了@objc为消息转发机制，和OC类中方法一样，可以在子类中重写，并且重写后的方法，同样是消息派发机制
    • 方法前加上@objc dynamic，swift和OC中调用，均为消息派发

静态派发

  • 内联
    • 编译器会自动将静态派发的某些函数变成内联函数
      • @inline,内联注解， Apple 的文档中是找不到，不建议使用
        • @inline(__always) func test(i:Int) { },开启编译优化后，即使代码很长也会被内联（递归调用,动态派发除外）
        • @inline(never) func test(i:Int) { },永远不会被内联，即使开启了编译器优化（有时不起作用,还是会转为内联）
  • 直接调用
    • 值类型、引用类型以及协议的extension中定义的方法，一般默认是静态派发
    • 值类型中定义的方法
    • 引用类型中用final修饰的方法
    • 所有用static修饰的方法

派发机制和重写的关系

  • 直接派发由于是直接跳到函数地址去执行，因此直接派发的方法是无法进行重写的
    • 类中extension中的方法，默认是直接派发，因此无法重写
    • extension中的方法，用@objc修饰会变成消息派发，子类可以重写
  • 函数派发和消息派发都可以由子类进行重写
    • 函数派发的重写，只能在子类的非extension中，也就是声明处
    • 消息派发的重写，是可以在子类的extension中进行重写的，类似OC中的分类
