objc_msgSend

  • OC中的方法调用，其实都是转换为objc_msgSend函数的调用
  • objc_msgSend的执行流程可以分为3大阶段
    • 消息发送
    • 动态方法解析
    • 消息转发

objc_msgSend源码(866)执行流程

objc-msg-arm64.s

  • objc-msg-arm64.s
    1. MSG_ENTRY _objc_msgSend
      • 到 END_ENTRY _objc_msgSend，objc_msgSend之间定义的。
    2. b.le LNilOrTagged
      • 首先会判断是否时nil，为nil，则会跳转到LReturnZero直接返回，
    3. CacheLookup NORMAL
      • 不是nil,去查找缓存
    4. .macro CacheLookup
      • 缓存命中CacheHit，直接返回IMP
    5. 缓存未命中，__objc_msgLookup_uncached
      • 没找到方法，则会调用MethodTableLookup去查找，MethodTableLookup
      • 最终调用的是C函数的方法是lookUpImpOrForward(obj, sel, cls, LOOKUP_INITIALIZE | LOOKUP_RESOLVER)（注意汇编中的比C函数方法多一个_）

第一阶段，消息发送

  • 在objc-runtime-new.mm文件中，第一个阶段，查找方法
    1. lookUpImpOrForward,为从缓存中查找的逻辑
    2. realizeAndInitializeIfNeeded_locked
      • 确保类已经realize了,初始化了
    3. for循环进行遍历查找类cls中方法，有个最大的数进行记录，防止陷入死循环
      1. 先去cache_getImp从缓存中获取
        • 在查找中间的方法过程中，可能会有新方法在中间添加进来，也都会先加到缓存中
      2. getMethodNoSuper_nolock、search_method_list_inline、log_and_fill_cache
        1. 从class_rw_t的methods中找到所有类和分类的方法，遍历method_list_t查找，因此需要类先realize
        2. 如果是排好序列的(排序的实现，是在类的realize方法中的methodizeClass进行解析的)，findMethodInSortedMethodList中去查找方法,根据方法名查找，已经排序，二分查找的方式
          • 如果有相同的有很多，会倒序查找，找到排序的第一个方法
          • 排序，会在prepareMethodLists中调用fixupMethodList方法
            • method_t中方法类型有big、small、bigSigned
              • big是默认的，大方法，其值为指针
              • small，小方法，其值表示为相对每个字段的额外地址，其结构是
                • 其引用的选择器在(共享缓存)或者是在磁盘上，考虑相关动态库的系统方法,该方法是只读内存，因此small的是无法修改的。其是否是small方法，是由编译器决定，进行设置的
              • bigSigned,和big相同，但是其name已经签名，该方法是由运行时添加的，类似class_addMethod方法添加的
            • 对方法排序，只会对非small的方法进行排序,排序是根据方法名SEL的地址由低到高排序的
            • 排序的方式采用的是**归并排序**
        3. 不是排好序的方法，调用findMethodInUnsortedMethodList，线性遍历查找，找到第一个方法名相同的，后，返回
        4. log_and_fill_cache(cls, imp, sel, inst, curClass),查找到方法后，调用void cache_t::insert,去插入到缓存中,保存的类是最开始查找的类方法cls中,而并非当前的类(即如果查找到了父类的IMP,最后会将这个IMP保存到子类的cache)。
        5. 当前curClass中没有找到，则会去父类遍历查找curClass = curClass->getSuperclass()) == nil
          • 如果找到最后的父类，都没有找到方法，则会设置当前imp = _objc_msgForward_impcache
          • 或者从缓存中取到的方法，等于_objc_msgForward_impcache,则会跳出循环
  • 消息发送流程

第二阶段，动态方法解析

  • log_and_fill_cache，第二个阶段，动态方法解析
    • (behavior & LOOKUP_RESOLVER) == 0由最开始汇编中，传入，需要进行方法解析，则进入方法解析流程
      1. behavior ^= LOOKUP_RESOLVER,解析之后，就不需要了
      2. resolveMethod_locked(inst, sel, cls, behavior),进入方法解析流程
      3. 不是元类，则走resolveInstanceMethod方法
        1. 会先去找resolveInstanceMethod,_lookUpImpOrNilTryCache(cls, resolve_sel, cls->ISA(true))先去,其isa，也就是其元类对象，缓存中找，找不到，则走阶段一的方法查找，这次的behavior不包含LOOKUP_RESOLVER,这次是不会去走动态方法解析
          • 其中NSObject中+(BOOL)resolveInstanceMethod会有基础实现
          • 因为后续会同步再次去查询方法，因此是否返回true还是false，不影响实际的结果，但是按规范，返回true
        2. 会用objc_msgSend调用方法
        3. 会再调用lookUpImpOrNilTryCache(inst, sel, cls),去查找，一开始寻找的方法
        4. 方法的返回，最后通过调用lookUpImpOrForwardTryCache去查找
          • 注意，用lookUpImpOrNilTryCache传入的behavior包括LOOKUP_NIL，即没有搜索到，会返回nil,不会出现找不到方法的异常
      4. 是元类，则会查找resolveClassMethod，同时，如果调用lookUpImpOrNilTryCache,没有找到，也会去走resolveInstanceMethod方法，即上面的流程(注意，此时他会去元类的isa指针中找方法，而元类的isa指针，也就是基元类，也就是NSObjec类方法中)，解析类的方法和实例方法流程一样
        1. 同样会先去找方法resolveClassMethod，其在NSObject中也已实现
        2. 会用objc_msgSend的方式调用
        3. 最终用lookUpImpOrForwardTryCache去查找方法

第三阶段，消息转发

  • 在第二阶段中，最终会用lookUpImpOrForwardTryCache,再次去查找一次方法，这次，如果动态解析方法添加到了，则会查找到方法，并直接返回；否则会将imp仍人为_objc_msgForward_impcache，返回。
  • 最终方法调用者会去执行_objc_msgForward_impcache,会跳转到该方法的实现上，该方法在objc-msg-arm64.s文件内，属于汇编
    • ENTRY __objc_msgForward
    • Core Foundation
    • __forwarding__（不开源)
  • 实际实现,类方法中也有消息转发处理
    1. -(id)forwardingTargetForSelector:返回可以转发该方法的对象，如果是类方法则是+
      • 注意，也可以在该实例方法中返回可以响应的类对象（如果类方法支持的话）
    2. 如果1中返回的是该对象本身，则会调用- (NSMethodSignature *)methodSignatureForSelector:(SEL)aSelector,先获取方法签名
      • 注意方法签名，会决定NSInvocation中的参数和返回值
    3. 2中获取了方法签名之后，再会调用forwardInvocation:(NSInvocation *)anInvocation方法
    4. 如果最后还是没有实现方法3，则会走doesNotRecognizeSelector 方法，发出未找到方法的错误
      • 注意，只要方法3实现了，就算没有任何实现，那么也不会走到doesNotRecognizeSelector的流程，要注意NSObjec中forwardInvocation:的默认实现,如果没实际实现转发的方法，还是会触发doesNotRecognizeSelector
  • 注意，消息转发成功之后，在当前调用的类中是不会缓存方法的，但是在转发的类中会缓存;因此,后续调用该方法，仍然无法从缓存中获取，每次调用该方法，都会重新走消息转发流程。

    总结

  • OC的消息机制
    • OC中的方法调用其实都是转成了objc_msgSend函数的调用，给receiver发送一条消息（selector方法）
    • objc_msgSend底层有3大阶段
      1. 消息发送（当前类、父类中查找）
      2. 动态方法解析
      3. 消息转发
