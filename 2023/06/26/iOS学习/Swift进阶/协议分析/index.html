协议方法调度

类的方法调度是通过函数表v-table的方式，那么类遵循协议的方法调度是如何的进行的

结论

  • 一个类实现了协议上的方法，并且创建这个类的对象时，
    1. 如果把这个对象声明成本身类的类型，会通过v-Table的方式调度函数；
    2. 如果把这个对象声明成协议类型，那么在对象调用实现协议的方法的时候会先去witness_table查找这个对象的具体类型和具体实现(v-Table)，并完成当前的方法调度。
  • 注意：
    1. 每一个遵循协议的类，都会为其创建witness_table,无论是扩展实现还是类实现
    2. 举例Person类遵循了ProtocolA并实现了协议方法，而Teacher类继承于Person类且没有遵循ProtocolA，Teacher类就不会创建witness_table
      • 尽管没有创建，当出现Teacher重写了Person中的协议方法时，用var t: MyProtocolA = Teacher()，调用重写的协议方法时，仍然是重写后的方法，而非父类方法。因为witness_table是将当前Teacher的数据转成Person之后再去调用的，而Teacher重写了方法，因此会调用重写后的方法
    3. Person类遵循了ProtocolA和ProtocolB并都实现了协议方法，它会为Person类分别创建两个Protocol Witness Table(PWT)

数据结构

Existential Container

Existential Container也就是本章所说的ProtocolBox实际上有两类,Opaque Existential Container和ClassExistentialContainer

Opaque Existential Container

Opaque Existential Container — 用于没有类约束的 Protocol (no class constraint on protocol)，也就是说这种协议背后的真实类型可能是类、结构体以及枚举等。因此其存储就非常复杂

  • fixedSizeBuffer:3 个指针大小的 buffer 空间，当真实类型的 size (内存对齐后的大小) 小于 3 个字时则其内容直接存储在 fixedSizeBuffer 中，否则在 heap 上另辟空间存储，并将指针存储在 fixedSizeBuffer 中；
  • type:指向真实类型的 Metadata，最重要的就是引用其中的 VWT 用于完成内存的各种操作；
  • witnessTables: 指向协议函数表 (Protocol Witness Table, PWT)，协议函数表中存储的是真实类型中对应函数的地址。

Class Existential Container

用于有类约束的 Protocol，该协议背后真实的类型只能是类，而类的实例都是在 Heap 上分配内存的。
因此，在 Existential Container 中只需要一个指向堆内存的指针即可

  • value:指向堆内存的指针；
  • witnessTables:PWT 指针。

注意点

继承AnyObject只用于类的协议，swift会进行优化,其内存占用不是之前的40字节，而变成16。

  • 如果协议只能用于类，例如协议继承了AnyObject，则协议的结构ProtocolBox会发生改变,其内存大小会变成16。

  • 此外，如果是BrigeProtocol.Type，协议的元类也是16字节

  • 如果协议能用于类、结构体、枚举，则协议的结构ProtocolBox会和此前分析的一致，其内存大小会变成40

总结

  1. 每个遵守了协议的类，都会有自己的Protocol Witness Table (PWT)，遵守的协议越多，PWT中存储的函数地址就越多；
  2. PWT的本质是一个指针数组，第一个成员存储TargetProtocolConformanceDescriptor，其后面存储的是实现协议函数的地址；
  3. PWT的数量与协议数量一致。

ProtocolBox的作用

  • Existential Container即ProtocolBox是编译器生成的一种特殊的数据类型，用于管理class/struct/enum等遵守了同一个协议类型，因为这些class/struct/enum等类型的内存大小不一致，所以通过当前的Existential Container统一管理；
  • 对于小容量的数据，直接存储在Value Buffer；(小于等于24字节),前三个8字节中
  • 对于大容量的数据，通过堆区分配，存储堆空间的地址。（大于24字节）
  • 将Circle换成struct
  • 查看打印的circle对象依旧是40个字节。
    • 因为当实例变量声明为协议类型的时候，程序在编译时并不能推导出实例变量的真实类型，那就导致没有办法为实例变量分配具体的内存空间。
    • 于是就不管它到底需要多大是什么类型统一塞到40字节的存在容器统一管理。
    • 所以编译器就用一种特殊的数据类型Existential Container去统一管理遵循协议的class/struct/enum等（Existential Container本质是上面分析出的ProtocolBox）
  • 前24个字节存放的数据,第四个8字节存放的是依旧是metadata,此时是TargetStructMetadata,在真实调用实现协议的方法的时候就会找到这个metadata

  • lldb查看,超过三个之后，需要把所有的内容移至堆区开辟一块内存空间去存放，ProtocolBox第一个成员就变成了一块堆区的地址，它存放了那五个成员变量的值。
